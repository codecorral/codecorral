## ADDED Requirements

### Requirement: Nix flake exposes engine package
The Nix flake SHALL expose the `codecorral` CLI as a package (`packages.${system}.codecorral`) that can be built and installed via Nix. The package SHALL build with proper dependency handling and complete without errors.

#### Scenario: Build engine package
- **WHEN** the user runs `nix build .#codecorral`
- **THEN** a result directory contains the `codecorral` binary with all dependencies bundled and the build completes without errors

#### Scenario: Install via Nix flake
- **WHEN** the user runs `nix profile install .#codecorral`
- **THEN** the `codecorral` binary is available on PATH and can execute all CLI commands

### Requirement: Home Manager module for project configuration
The Nix flake SHALL expose a Home Manager module (`homeManagerModules.codecorral`) that generates `~/.codecorral/config.yaml` from declarative Nix expressions and delegates tool-specific configuration to upstream Nix modules. The module SHALL use `programs.codecorral.enable` as its activation guard. The module SHALL require agentplot-kit's claude-code HM module (not the upstream nix-community one) for profile support. The generated config file SHALL be valid YAML (not JSON) and SHALL contain only engine-own state.

The module SHALL delegate per-project settings:
- `programs.codecorral.projects.<name>.claude_code` passes through to `programs.claude_code.profiles.<name>.*`
- Each project auto-creates an agent-deck profile with `claude.config_dir` convention
- OpenSpec schemas are collected as a union across all projects

Global agent-deck configuration (conductor, MCPs, shell, docker) is the user's responsibility via `programs.agent-deck` directly. Conductor/session layout will be handled by the `configure-agent-deck-session` intent via shuffle in a future change.

#### Scenario: HM module generates config.yaml with engine-own state only
- **WHEN** the user declares:
  ```nix
  programs.codecorral.enable = true;
  programs.codecorral.projects.my-project = {
    path = "/path/to/project";
    workflows = [ "intent" "unit" ];
    claude_code.settings.model = "claude-sonnet-4-6";
    openspec.schemas = [ "dev.codecorral.intent@2026-03-11.0" ];
  };
  ```
- **THEN** `home-manager switch` generates `~/.codecorral/config.yaml` containing:
  ```yaml
  # Generated by CodeCorral Home Manager module
  projects:
    my-project:
      path: /path/to/project
      workflows:
        - intent
        - unit
      agent_deck_profile: my-project
  ```
  Claude-code model and openspec schemas are NOT in config.yaml — they are delegated to upstream modules. Global agent-deck settings are configured directly via `programs.agent-deck`.

#### Scenario: Generated config is valid YAML
- **WHEN** the HM module generates `~/.codecorral/config.yaml`
- **THEN** the file is valid, human-readable YAML (not JSON serialized via `builtins.toJSON`)

#### Scenario: Per-project agent-deck profile creation
- **WHEN** a project is declared
- **THEN** the module creates `programs.agent_deck.profiles.<project_name>.claude.config_dir` set to `.claude-<project_name>` via `lib.mkDefault` (user can override)

#### Scenario: Per-project claude-code full pass-through
- **WHEN** a project declares `claude_code` with settings, agents, rules, skills, or other options
- **THEN** the module passes all settings through to `programs.claude_code.profiles.<project_name>.*` and auto-sets `config_dir = ".claude-<project_name>"` via `lib.mkDefault`

#### Scenario: Shared config_dir convention
- **WHEN** a project is declared without explicit `config_dir` overrides
- **THEN** both `programs.agent_deck.profiles.<name>.claude.config_dir` and `programs.claude_code.profiles.<name>.config_dir` are `.claude-<project_name>`

#### Scenario: User overrides config_dir
- **WHEN** a project sets `claude_code.config_dir = ".claude-custom"`
- **THEN** the explicit value wins over the `lib.mkDefault` convention

#### Scenario: OpenSpec schema union
- **WHEN** multiple projects declare `openspec.schemas`
- **THEN** the module sets `programs.openspec.schemas` to the union of all project schema lists

#### Scenario: Upstream agent-deck module not imported
- **WHEN** `programs.agent_deck` is not available in the user's HM config
- **THEN** per-project agent-deck profile creation is skipped without error

#### Scenario: Upstream claude-code module lacks profiles
- **WHEN** the user has the upstream nix-community claude-code module (no `profiles` option) instead of agentplot-kit
- **THEN** claude-code delegation is skipped without error; a warning is logged

#### Scenario: Duplicate profile names rejected
- **WHEN** two projects would produce the same profile name
- **THEN** `home-manager switch` fails with an assertion error

### Requirement: npm package for non-Nix installation
The engine SHALL be published as an npm package with a `bin` entry for `codecorral`. Users SHALL be able to run `npx codecorral` or install globally via `npm install -g codecorral`.

#### Scenario: Run via npx
- **WHEN** the user runs `npx codecorral status`
- **THEN** the CLI executes without requiring Nix, Home Manager, or any other CodeCorral infrastructure

#### Scenario: Global npm install
- **WHEN** the user runs `npm install -g codecorral` followed by `codecorral status`
- **THEN** the CLI is available globally and connects to or starts the engine daemon

### Requirement: Consistent behavior across installation methods
The `codecorral` CLI SHALL behave identically whether installed via Nix or npm. The only difference SHALL be configuration: Nix users get declarative config via HM module, npm users write `~/.codecorral/config.yaml` manually.

#### Scenario: Same CLI output from both install methods
- **WHEN** the same command is run from a Nix-installed and npm-installed `codecorral` against the same daemon
- **THEN** the output is identical
